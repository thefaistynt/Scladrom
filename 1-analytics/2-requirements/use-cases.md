# Use Cases — скалодром «Вертикаль» (клиентское мобильное приложение)

> **Источник:** [domain.md](../1-elicitation/domain.md)
> **Приоритеты:** MoSCoW — **Must** (обязательно для MVP), **Should** (желательно), **Could** (post-MVP), **Won't** (вне скоупа)

### UC-001. Регистрация клиента
- **Актор:** Клиент
- **Предусловие:** пользователь не зарегистрирован в системе
- **Основной сценарий:**
  1. Клиент открывает приложение и переходит к регистрации.
  2. Вводит email и пароль.
  3. Система подтверждает совершеннолетие (18+) и создаёт учётную запись.
- **Альтернативный сценарий:** возраст/данные не подтверждены — регистрация отклоняется.
- **Приоритет:** Must
- **Источник:** §4.1; §5.1; BR-011

### UC-002. Просмотр расписания слотов
- **Актор:** Клиент
- **Предусловие:** клиент авторизован
- **Основной сценарий:**
  1. Клиент открывает раздел расписания.
  2. Система показывает слоты на 7 дней вперёд.
  3. Клиент применяет фильтры (формат, инструктор, время суток, свободные места).
  4. Система обновляет список слотов.
- **Альтернативный сценарий:** подходящих слотов нет — система показывает empty state «Пока нет доступных тренировок».
- **Приоритет:** Must
- **Источник:** §4.2; §5.2; BR-005

### UC-003. Бронирование тренировки
- **Актор:** Клиент
- **Предусловие:** клиент авторизован, выбран слот
- **Основной сценарий:**
  1. Клиент выбирает слот, указывает количество мест.
  2. Клиент выбирает: своё снаряжение или полный прокат.
  3. Система показывает правила отмены и итоговую стоимость (тренировка + прокат отдельной строкой).
  4. Клиент подтверждает запись.
  5. Бэкенд атомарно проверяет доступность мест и создаёт бронирование со статусом «Активна».
- **Альтернативные сценарии:**
  - 3a. Прокатного фонда недостаточно → система показывает предупреждение; клиент подтверждает — бронирование создаётся.
  - 5a. Место занято (конфликт) → система показывает сообщение «место уже занято»; бронирование не создаётся.
  - Допуск по уровню: новичок не может забронировать формат «опытный» (проверка перед подтверждением).
- **Приоритет:** Must
- **Источник:** §4.3; §4.4; §5.3; BR-004, BR-006, BR-012–BR-016

### UC-004. Отмена бронирования клиентом
- **Актор:** Клиент
- **Предусловие:** есть активное бронирование
- **Основной сценарий:**
  1. Клиент открывает бронирование и инициирует отмену.
  2. Система проверяет, что до начала тренировки ≥24 часа.
  3. Бронирование переходит в статус «Отменена клиентом».
- **Альтернативный сценарий:** до начала <24 часов → кнопка отмены недоступна, отмена невозможна (в т.ч. в случае неявки — отдельного сценария no-show нет).
- **Приоритет:** Must
- **Источник:** §4.3; §5.4; BR-009, BR-018

### UC-005. Получение уведомления об отмене тренировки скалодромом
- **Актор:** Клиент (инициатор отмены — Владелец/администратор, вне клиентского приложения)
- **Предусловие:** у клиента есть активное бронирование на слот, который отменяет администратор
- **Основной сценарий:**
  1. Администратор отменяет слот в существующей инфраструктуре (вне клиентского приложения).
  2. Бронирование клиента переходит в статус «Отменена скалодромом» с указанием причины.
  3. Клиент получает push-уведомление с извинениями.
- **Постусловие:** повторная запись клиента на этот же слот запрещена; переноса не предусмотрено.
- **Приоритет:** Must
- **Источник:** §5.5; BR-007, BR-008

### UC-006. Оценка инструктора (post-MVP)
- **Актор:** Клиент
- **Предусловие:** тренировка завершена не более 24 часов назад
- **Основной сценарий:**
  1. Клиент открывает форму оценки завершённой тренировки.
  2. Указывает оценку (1–5) и текстовый отзыв.
  3. Отправляет оценку — данные становятся доступны только владельцу (в аналитике).
- **Альтернативный сценарий:** окно 24 часа истекло — оценка недоступна.
- **Ограничение:** после отправки оценка не редактируется.
- **Приоритет:** Could (post-MVP)
- **Источник:** §4.5; §5.7
